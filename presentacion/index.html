// 01

Fundamentos del Diseño de Agentes

Un agente IA es una unidad autónoma con un propósito claro, herramientas específicas y un contexto bien definido.

🎯
Principio de Responsabilidad Única
Cada agente hace una sola cosa muy bien. Un agente que "hace todo" falla en todo. La especialización permite entrenamiento de contexto preciso, prompts más cortos y resultados más predecibles.
🧩
Composición sobre Monolitos
Varios agentes pequeños conectados superan a uno grande. Cada uno puede ser reemplazado, mejorado o reordenado sin romper el sistema. Piensa en microservicios, no en monolitos.
📡
Contratos de Comunicación
Define cómo los agentes se pasan información: formato de input, formato de output, errores esperados. Sin un contrato claro, la cadena se rompe.
🔁
Idempotencia
Un agente bien diseñado puede correr múltiples veces con el mismo input y producir el mismo output. Esto es esencial para debugging y reintentos.
📝
Memoria & Estado
Decide conscientemente qué información persiste entre llamadas (memoria larga) y qué es solo del turno actual (contexto de sesión). No mezcles ambas.
🛑
Fail-Safe & Escalamiento
Define qué pasa cuando un agente falla: ¿reintenta? ¿avisa al orquestador? ¿pausa el pipeline? Diseña el fallo antes de diseñar el éxito.
// Regla de oro

Un agente debe ser describible en una sola oración. Si necesitas "y" para describirlo, tenés dos agentes.

Los 4 tipos de agentes más comunes

Tipo A
Orquestador / Planner
Recibe el objetivo general, lo descompone en subtareas y delega a otros agentes. No ejecuta, coordina.
Tipo B
Executor / Worker
Recibe una tarea específica y la ejecuta usando herramientas. No piensa, hace.
Tipo C
Reviewer / Critic
Evalúa el output de otros agentes contra criterios definidos. No crea, juzga.
Tipo D
Specialist
Experto en un dominio muy específico (N8N, código Python, diseño). Profundidad, no amplitud.

Anatomía de un Agente

Los 5 componentes que definen cualquier agente, independientemente de la tarea.

ESTRUCTURA BASE DE UN AGENTE
// YAML de configuración conceptual
agent:
  id: "n8n-builder-v1"          # Identificador único
  role: "N8N Workflow Architect"   # Un título descriptivo
  version: "1.0"

  identity:                       # Quién es el agente
    persona: "Sos un experto en..."
    tone: "técnico, conciso"

  objective:                      # Qué hace
    primary: "Diseñar workflows N8N..."
    scope: "Solo N8N, no otros sistemas"

  context:                        # Qué sabe
    domain_knowledge: ["n8n-docs", "api-patterns"]
    constraints: ["solo v1.x", "sin credenciales hardcoded"]

  tools:                          # Qué puede usar
    - read_file
    - write_file
    - search_docs

  io:                            # Contratos de entrada/salida
    input_format: "descripción en lenguaje natural"
    output_format: "JSON de nodos N8N"
    error_format: "{ error: string, suggestions: [] }"

🧠 Identity (quién es)

01
Persona clara
Define el rol con experiencia específica. No "asistente IA", sino "arquitecto de workflows con 5 años en N8N".
02
Tono consistente
Técnico, formal, casual, conciso. El tono afecta el formato del output.
03
Límites de identidad
Qué NO es: "No soy un asistente general. Solo resuelvo problemas de N8N."

📋 Objective (qué hace)

A
Objetivo primario
Una tarea central en una oración. El agente debe poder decir "mi trabajo es X".
B
Scope explícito
Qué entra y qué sale de su responsabilidad. El scope previene scope creep.
C
Criterio de éxito
¿Cómo sabe el agente que terminó bien? Define el "done" explícitamente.
// 03

Cómo Separar Agentes

La separación correcta es la decisión más importante. Hay 3 ejes para decidir cuándo un agente es uno o dos.

🗂️
Por Dominio de Conocimiento
Separá agentes que necesitan conocimiento de contextos distintos.

N8N Agent Python Agent Slides Agent

No mezcles: un agente que "sabe N8N y Python y diseño" va a ser mediocre en todo.
⚡
Por Tipo de Operación
Separá agentes según el tipo de acción que realizan.

Planner Executor Tester Documenter

Planear y ejecutar son capacidades mentalmente distintas, requieren prompts distintos.
🔄
Por Frecuencia de Uso
Si una función se usa mucho más que otras, merece su propio agente.

Si el 80% de tus tareas son de código, un agente de código generalista tiene sentido. Las tareas de presentaciones van a un agente separado.
📏
Por Tamaño de Contexto
Cuando un agente necesita demasiado contexto para hacer su trabajo, dividilo.

Un agente que necesita saber N8N docs + Python + git + diseño = ventana de contexto saturada = resultados pobres.

Hacete estas preguntas: ¿El agente necesita saber cosas completamente distintas para partes distintas del trabajo? ¿El prompt supera 500 tokens solo de contexto? ¿El agente a veces hace A y a veces hace B, nunca ambas juntas? → Son dos agentes.

// 04

Contexto & Estructura de Prompts

El contexto es el conocimiento estático del agente. El prompt es la instrucción dinámica. Nunca los mezcles.

📚
Contexto (CLAUDE.md)
Es el system prompt. Define quién es el agente, qué sabe y cómo opera.

Cambia raramente. Es la "configuración" del agente.

Incluye: persona, dominio, restricciones, formato de output, ejemplos de patrones.
✉️
Prompt (user message)
Es la instrucción de la tarea actual. Específica, acotada, con el estado actual.

Cambia en cada llamada. Es el "qué hacer ahora".

Incluye: tarea específica, datos de entrada, resultado esperado, contexto de esta tarea.

Estructura de CLAUDE.md (System Prompt)

// agents/n8n-agent/CLAUDE.md
# ROL Y PERSONA
Sos un experto en N8N con dominio profundo de workflows de automatización.
Tu especialidad: integrar APIs REST, transformar datos y diseñar flujos robustos.
Tono: técnico y directo. Sin explicaciones innecesarias.

# OBJETIVO
Generar configuraciones de workflows N8N en formato JSON válido, listo para importar.
Scope: Solo N8N. Si la tarea requiere código Python o diseño, declinás y lo indicás.

# CONOCIMIENTO DE DOMINIO
- Versión target: N8N 1.x
- Nodos disponibles: [lista de nodos relevantes]
- Patrones aprobados: webhook → transform → send, scheduler → fetch → store
- Restricciones: no hardcodear credenciales, usar variables de entorno

# FORMATO DE OUTPUT
Siempre respondé con:
1. Breve descripción del workflow (2-3 líneas)
2. JSON de configuración entre ```json ``` 
3. Lista de variables de entorno necesarias

# MANEJO DE ERRORES
Si la tarea está fuera de scope: "OUT_OF_SCOPE: [razón]"
Si falta información: "NEEDS_INFO: [qué información necesitás]"

# EJEMPLOS DE REFERENCIA
[Ejemplos de workflows bien estructurados para que el agente aprenda el patrón]

Estructura del Prompt de Usuario

// Plantilla de prompt para llamar al agente
# TAREA
[Descripción clara de qué hacer en esta llamada específica]

# INPUT
[Datos de entrada estructurados - JSON, texto, archivo]

# CONTEXTO DE ESTA TAREA
[Información relevante SOLO para este run: estado anterior, decisiones tomadas, etc.]

# OUTPUT ESPERADO
[Qué formato y contenido necesitás del output]

# RESTRICCIONES ADICIONALES
[Cualquier limitación específica de esta tarea que no está en el system prompt]
// Tip: Few-shot en el contexto

Incluir 2-3 ejemplos de input/output en el CLAUDE.md es lo que más mejora la calidad. El modelo aprende el patrón exacto que querés. No expliques cómo hacerlo, mostrá cómo se ve cuando está bien hecho.

Los 3 niveles de contexto

L1
Contexto Global (compartido entre todos los agentes)
Stack tecnológico, convenciones de nomenclatura, estilo de código del proyecto, quién usa el sistema. Va en _shared/base-context.md e importado en cada CLAUDE.md.
L2
Contexto del Agente (específico del rol)
Conocimiento de dominio, patrones específicos, ejemplos. Va en el CLAUDE.md del agente. Solo este agente lo necesita.
L3
Contexto de Sesión (específico de esta tarea)
Estado del pipeline, resultado del agente anterior, decisiones de esta sesión. Va en el prompt de usuario. Se descarta después.

Pipeline Multi-Agente

El patrón Planner → Executor → Tester → Documenter: cómo conectarlos y qué pasa entre cada etapa.

🧭
Agente 1
Planner
Descompone el objetivo en subtareas
⚡
Agente 2
Executor
Ejecuta cada subtarea
🔍
Agente 3
Tester
Evalúa y valida el output
📄
Agente 4
Documenter
Genera docs del resultado
// Contrato de comunicación entre agentes
/* Planner → Executor */
{
  "plan_id": "task-2024-001",
  "objective": "Crear workflow N8N para sincronizar leads",
  "subtasks": [
    { "id": "t1", "agent": "n8n-agent", "task": "Crear nodo webhook", "depends_on": [] },
    { "id": "t2", "agent": "n8n-agent", "task": "Agregar transform", "depends_on": ["t1"] }
  ],
  "context": { "api_docs": "...", "credentials_ref": "env.CRM_KEY" }
}

/* Executor → Tester */
{
  "plan_id": "task-2024-001",
  "status": "completed",
  "artifacts": { "workflow_json": "...", "env_vars": ["CRM_KEY"] },
  "notes": "Usé HTTP Request node porque Webhook tiene limitaciones en v1.2"
}

/* Tester → Documenter */
{
  "plan_id": "task-2024-001",
  "test_result": "PASSED",
  "issues": [],
  "approved_artifacts": { "workflow_json": "..." },
  "test_notes": "Validado contra 3 casos de prueba"
}

Patrones de orquestación

⛓️
Chain (Cadena Lineal)
A → B → C → D

El output de cada agente es el input del siguiente. Simple y predecible. Ideal para pipelines donde cada paso depende del anterior.
🌲
Tree (Árbol de Tareas)
Planner descompone en subtareas paralelas que distintos ejecutores resuelven, luego un merger une los resultados. Ideal para tareas independientes.
🔄
Loop (Ciclo de Refinamiento)
Executor → Reviewer → (si falla) → Executor. El agente cicla hasta que el reviewer aprueba. Máximo N iteraciones para evitar loops infinitos.
🎯
Router (Enrutador)
Un agente clasificador decide qué agente especialista activa. Ideal cuando no sabés de antemano qué tipo de tarea vas a recibir.

Skills & Herramientas

Las skills son capacidades concretas que le das al agente. Las herramientas son lo que el agente puede invocar.

📁 Skills como archivos de conocimiento

Una skill es un documento de referencia que el agente puede consultar. No es código ejecutable, es conocimiento estructurado.

skills/
├── n8n-patterns.md      # Patrones N8N
├── api-integration.md   # Cómo integrar APIs
├── code-review.md       # Checklist de revisión
├── slide-design.md      # Principios visuales
└── documentation.md     # Estilo de docs

🔧 Herramientas (tools) del agente

Lo que el agente puede hacer en el sistema. Dáselas con principio de mínimo privilegio.

Herramientas comunes:

read_file       # Leer archivos del proyecto
write_file      # Escribir/crear archivos
search_files    # Buscar en el proyecto
run_command     # Ejecutar comandos shell
web_search      # Buscar en internet
call_api        # Llamar APIs externas

Cómo referenciar skills en el CLAUDE.md

// En el CLAUDE.md del agente
# SKILLS DISPONIBLES
Antes de ejecutar cualquier tarea, consultá los siguientes archivos de referencia:

- @skills/n8n-patterns.md  → patrones de workflows aprobados
- @skills/api-integration.md → cómo estructurar llamadas API

# CUÁNDO USAR CADA SKILL
- Si la tarea implica crear un workflow: leer n8n-patterns.md primero
- Si hay una integración de API externa: leer api-integration.md
- Si hay duda sobre el enfoque: buscar en ambos antes de preguntar

Asignación de herramientas por tipo de agente

🧭 Planner
read_file search_files web_search

El planner solo necesita leer e investigar. No escribe ni ejecuta.
⚡ Executor
write_file run_command call_api

El executor necesita actuar. Tiene herramientas de escritura y ejecución.
🔍 Tester
read_file run_command write_file

El tester lee, ejecuta tests y escribe el reporte. Scope acotado de escritura.
📄 Documenter
read_file write_file

El documenter solo lee el resultado final y escribe la documentación. Mínimas herramientas.

Plantillas Listas para Usar

CLAUDE.md base para cada tipo de agente. Copiá, personalizá, deploy.

🧭 Planner Agent — CLAUDE.md ▶
// agents/planner/CLAUDE.md
# ROL
Sos un Arquitecto de Tareas. Tu trabajo es analizar objetivos complejos
y descomponerlos en subtareas ejecutables, claras y con dependencias explícitas.

# OBJETIVO
Dado un objetivo en lenguaje natural, producir un plan estructurado JSON
que otros agentes puedan ejecutar sin ambigüedad.

# PROCESO
1. Identificar el objetivo principal
2. Listar todos los pasos necesarios
3. Identificar dependencias entre pasos
4. Asignar cada paso al agente correcto
5. Estimar complejidad (low/medium/high)

# AGENTES DISPONIBLES
- "n8n-agent": para workflows N8N
- "code-agent": para scripts Python/JS
- "slides-agent": para presentaciones
- "reviewer-agent": para validación

# OUTPUT FORMAT
{
  "plan_id": "uuid",
  "objective": "string",
  "estimated_complexity": "low|medium|high",
  "subtasks": [
    {
      "id": "t1",
      "agent": "agent-name",
      "task": "descripción clara",
      "expected_output": "qué debe producir",
      "depends_on": [],
      "context_needed": {}
    }
  ]
}

# RESTRICCIONES
- No ejecutes ninguna tarea, solo planificá
- Si el objetivo es ambiguo, listá las asunciones que hacés
- Máximo 10 subtareas por plan; si hay más, creá sub-planes
⚡ N8N Executor Agent — CLAUDE.md ▶
// agents/n8n-agent/CLAUDE.md
# ROL
Sos un especialista en N8N con conocimiento profundo de todos sus nodos,
integraciones y patrones de workflows de automatización empresarial.

# OBJETIVO
Generar configuraciones JSON de workflows N8N válidas e importables,
basadas en requisitos funcionales en lenguaje natural.

# CONOCIMIENTO DE DOMINIO
- N8N versión target: 1.x
- Patrones aprobados: @skills/n8n-patterns.md
- No hardcodear credenciales: siempre usar {{ $env.VARIABLE_NAME }}
- Preferir nodos nativos sobre HTTP Request cuando existen

# OUTPUT FORMAT
Siempre:
1. Descripción del workflow (3 líneas máx)
2. ```json (workflow completo)```
3. Lista de env vars requeridas
4. Instrucciones de activación

# MANEJO DE SCOPE
- OUT_OF_SCOPE si la tarea requiere código Python, diseño, o documentación
- NEEDS_INFO si faltan: URL de API, estructura de datos, credenciales ref
🔍 Reviewer / Tester Agent — CLAUDE.md ▶
// agents/reviewer-agent/CLAUDE.md
# ROL
Sos un Revisor Crítico. Tu trabajo es evaluar outputs de otros agentes
contra criterios de calidad definidos, sin ejecutar ni modificar el trabajo.

# OBJETIVO
Dado un artifact (código, workflow, documento), producir un veredicto
estructurado con issues específicos y nivel de severidad.

# CRITERIOS DE EVALUACIÓN
Leer siempre: @skills/review-checklist.md

Evaluar:
- Corrección funcional (¿hace lo que dice?)
- Seguridad (¿expone credenciales? ¿inputs sin validar?)
- Mantenibilidad (¿es legible? ¿tiene comentarios?)
- Consistencia con estándares del proyecto

# OUTPUT FORMAT
{
  "verdict": "APPROVED | REJECTED | NEEDS_REVISION",
  "score": 0-10,
  "issues": [
    {
      "severity": "CRITICAL|HIGH|MEDIUM|LOW",
      "location": "dónde está el problema",
      "description": "qué está mal",
      "suggestion": "cómo corregirlo"
    }
  ],
  "approved_for_next_stage": true|false
}

# RESTRICCIONES
- No modifiques el artifact, solo evaluá
- Si es CRITICAL: verdict = REJECTED automáticamente
- Sé específico: no "hay un bug", sino "línea X hace Y cuando debería Z"
📄 Documenter Agent — CLAUDE.md ▶
// agents/documenter-agent/CLAUDE.md
# ROL
Sos un Technical Writer especializado en documentación clara y accionable.
Tu audiencia: desarrolladores que necesitan usar el sistema, no construirlo.

# OBJETIVO
Dado un artifact aprobado y notas del proceso, generar documentación
técnica completa en Markdown, lista para publicar.

# ESTRUCTURA ESTÁNDAR
1. Descripción (qué hace, en 1 párrafo)
2. Requisitos previos
3. Configuración (paso a paso)
4. Uso (ejemplos concretos)
5. Variables de entorno (tabla)
6. Troubleshooting (los 3-5 errores más comunes)
7. Changelog (si aplica)

# ESTILO
- Verbos en imperativo ("Configurá", "Ejecutá")
- Ejemplos reales, no placeholders vagos
- Tablas para variables y parámetros
- Code blocks con syntax highlighting

# INPUT ESPERADO
{
  "artifact": {...},           // El resultado del executor
  "review_notes": {...},       // Notas del tester
  "process_context": "..."     // Por qué se hicieron ciertas decisiones
}

Recomendaciones Clave

Las lecciones más importantes para diseñar agentes que realmente funcionen.

01
Versionar los CLAUDE.md como código
Usá git para trackear cambios en los system prompts. Un cambio en el CLAUDE.md es un cambio de comportamiento del agente. Tratalos con el mismo rigor que el código fuente.
02
Definir el output format primero
Antes de escribir el system prompt, decidí exactamente qué formato de salida necesitás. El output format determina casi todo lo demás del diseño del agente.
03
Empezar con un agente, no con diez
Construí y refiná un agente hasta que funcione bien antes de agregar más. La tentación de diseñar el sistema completo de entrada lleva a complejidad prematura.
04
Loguear las llamadas entre agentes
Guardá en archivos el input y output de cada agente en cada corrida. Sin logs, es imposible debuggear dónde se rompió el pipeline. Un archivo JSON por run es suficiente.
05
Temperatura baja para agentes de ejecución
Para executors y reviewers, usá temperature 0 o 0.1. Para planners o agentes creativos, podés subir a 0.3-0.5. La consistencia es más valiosa que la creatividad en la mayoría de los casos.
06
No pasar contexto que el agente no usa
Cada token de contexto que el agente no necesita es ruido. El tester no necesita el plan original completo, solo el artifact que debe evaluar. Contexto mínimo, foco máximo.
07
Diseñar para el fallo desde el inicio
Define qué hace el pipeline si un agente devuelve un error. ¿Reintenta? ¿Notifica? ¿Pausa y espera input humano? El happy path es fácil, el manejo de errores es donde viven los bugs.
08
Human-in-the-loop en decisiones críticas
Para tareas irreversibles (deploy, envío de emails, modificación de base de datos), agregá un checkpoint humano antes de que el executor actúe. El agente presenta el plan, vos aprobás.
09
Usar ejemplos (few-shot) en vez de instrucciones largas
3 ejemplos de input/output bien elegidos superan a 500 palabras de instrucciones. Si tenés dificultades para que el agente entienda el formato, agregá un ejemplo antes de agregar más texto.
10
Naming consistente para los agentes
Usá un patrón: [dominio]-[tipo]-v[version]. Ejemplo: n8n-executor-v1, code-reviewer-v2. Cuando tengas 20 agentes, vas a agradecer la consistencia.

Checklist antes de deployar un agente

✅ CLAUDE.md tiene...
□ Rol y persona clara
□ Objetivo en una oración
□ Scope explícito (incluye y excluye)
□ Formato de output definido
□ Al menos 1-2 ejemplos (few-shot)
□ Manejo de errores / OUT_OF_SCOPE
□ Referencias a skills relevantes
✅ El pipeline tiene...
□ Contratos de input/output entre agentes
□ Logging de cada llamada
□ Manejo de errores definido
□ Máximo de reintentos configurado
□ Checkpoint humano en puntos críticos
□ Tests con casos de ejemplo
□ Documentación del flujo completo
